# 桌面数据分析器第一版设计说明

- 日期：2026-05-02
- 主题：轻量化本地 Excel 数据分析桌面软件第一版
- 目标：先做出可运行、可见、可演示的最小成品源码版，为后续打包 `.exe` 做准备

## 1. 产品定位

第一版产品定位为：**专业单页工作台式桌面软件**。

它不是脚本工具外壳，也不是分步骤向导，而是一个打开后即可在同一主界面中完成导入、处理、预览、图表查看和导出的桌面应用。

设计目标：
- 看起来像正式软件，而不是临时工具
- 本地运行、本地处理，不依赖云端
- 保持轻量，避免复杂但暂时无必要的高级能力
- 先交付可运行源码版，确认效果后再推进 `.exe` 打包交付

## 2. 为什么选择这个方案

围绕当前需求，有三种常见方向：

### 方案 A：专业单页工作台
将数据导入、处理配置、结果预览、图表展示都放在一个主窗口中。

优点：
- 最接近正式商业软件的观感
- 操作链路连续，适合频繁试错和查看结果
- 后续增加配置项、更多图表或导出能力时扩展自然

缺点：
- 第一版实现量比向导式略高

### 方案 B：步骤向导式
用户按步骤依次完成导入、处理、查看结果。

优点：
- 更容易理解
- 第一版开发简单

缺点：
- 产品感偏弱
- 后续扩展容易受限

### 方案 C：极简工具壳
只有几个按钮、一个表格、一个图表区域，快速把功能堆出来。

优点：
- 最快看到结果

缺点：
- 不像正式交付软件
- 后续大概率要重构界面和结构

最终选择：**方案 A：专业单页工作台**。

## 3. 第一版范围

第一版只做需求文档中明确提到、且可以合理形成最小成品的能力。

### 3.1 必做能力
- 读取本地 Excel 文件
- 展示原始数据预览
- 执行基础数据整理和加工
- 展示处理后的结果表格
- 展示基础可视化图表
- 导出处理结果

### 3.2 第一版内置的数据处理能力
由于 `项目描述.md` 没有给出精细业务规则，第一版采用通用、稳妥的本地分析处理能力：
- 去除全空行
- 去除重复行
- 按选定列排序
- 按选定列进行基础筛选
- 新增计算列
- 对数值列进行基础聚合或统计展示所需准备

### 3.3 第一版图表能力
第一版先支持最常见、最稳妥的图表：
- 柱状图
- 折线图

图表生成依赖用户选择的分类列与数值列。

### 3.4 暂不纳入第一版的内容
以下内容不进入第一版，以避免范围失控：
- 复杂多步骤流程编排
- 拖拽式数据处理节点
- 高级公式语言设计
- 多文件批处理任务系统
- 云同步、在线协作、账号系统
- 大而全 BI 能力

## 4. 用户界面设计

### 4.1 总体布局
采用单窗口工作台结构：

- **顶部工具栏**
  - 导入 Excel
  - 执行处理
  - 导出结果
  - 重置当前工作区

- **左侧配置区**
  - 数据源信息
  - 处理选项
  - 排序与筛选配置
  - 计算列配置
  - 图表字段选择

- **中间主区域**
  - 表格预览区
  - 支持切换查看“原始数据”和“处理结果”

- **右侧图表区**
  - 显示柱状图或折线图
  - 在字段选择变化后重新生成图表

### 4.2 交互流程
第一版典型流程如下：
1. 用户导入本地 Excel 文件
2. 软件读取首个工作表并显示原始数据预览
3. 用户在左侧选择是否去空行、去重、排序、筛选和新增计算列
4. 用户点击执行处理
5. 软件生成处理结果并展示在中间区域
6. 用户选择图表字段并查看图表
7. 用户导出结果文件

## 5. 技术设计

### 5.1 技术目标
- 本地 Python 实现
- 先以源码可运行版本为目标
- 代码结构直接考虑未来打包为 Windows `.exe`
- UI、数据处理、文件读写分层，避免后续难以维护

### 5.2 推荐技术方案
在当前信息不足但要求“尽快看到正式效果”的前提下，推荐：

- **桌面 UI**：PySide6
  - 原因：界面效果更接近正式桌面软件，控件能力完整，后续扩展性比 Tkinter 更好
- **数据处理**：pandas
  - 原因：处理 Excel、筛选、排序、去重、计算列都成熟直接
- **Excel 读取/导出**：pandas + openpyxl
- **图表**：PySide6 内嵌 matplotlib
  - 原因：稳定、成熟、可快速完成折线图和柱状图展示
- **后续打包**：PyInstaller
  - 原因：适合 Windows 单文件或目录式打包，是后续 `.exe` 交付的自然路径

### 5.3 建议模块划分
建议从一开始按职责拆分：

- `app/`：应用入口和启动逻辑
- `ui/`：主窗口、布局、控件与事件绑定
- `services/excel/`：Excel 读取与导出
- `services/transform/`：数据处理逻辑
- `services/chart/`：图表生成逻辑
- `models/`：界面和处理过程中的数据模型
- `tests/`：后续测试

## 6. 数据处理规则设计

在缺少详细业务规则的情况下，第一版采用清晰、可解释的规则：

### 6.1 数据读取
- 默认读取用户选择文件中的第一个工作表
- 第一行默认作为表头
- 读取后将表格转为 DataFrame

### 6.2 清洗规则
- 去空行：删除所有单元格均为空的行
- 去重：删除整行完全重复的数据

### 6.3 排序规则
- 用户选择一个列作为排序列
- 用户选择升序或降序

### 6.4 筛选规则
- 用户选择一个列
- 用户输入关键词
- 先使用包含匹配作为第一版规则

### 6.5 计算列规则
- 用户输入新列名称
- 用户选择已有数值列
- 第一版支持基础运算：加、减、乘、除

### 6.6 导出规则
- 处理结果导出为新的 Excel 文件
- 默认不覆盖原文件，输出为新文件名

## 7. 错误处理设计

第一版需要处理的主要错误场景：
- 文件无法打开
- Excel 内容为空或结构异常
- 选择的列不存在
- 计算列输入不完整
- 图表字段选择不合法
- 导出失败

处理原则：
- 在界面上直接给出可读提示
- 不让程序崩溃退出
- 对当前操作失败进行明确反馈

## 8. 测试与验收

### 8.1 第一版最小验收标准
满足以下条件即可认为第一版可演示：
- 能启动桌面界面
- 能导入一个常规 Excel 文件
- 能看到原始数据表格
- 能执行至少一组基础处理
- 能看到处理结果表格
- 能生成至少一种图表
- 能导出处理后的结果文件

### 8.2 建议最先验证的样例
建议用一个包含以下列的简单 Excel 样表进行验证：
- 日期列
- 分类列
- 数值列
- 少量空值和重复值

这样可以同时验证读取、清洗、排序、筛选、计算和图表能力。

## 9. 实施边界

后续进入编码阶段时，应优先保证：
- 主流程可跑通
- 界面完整度优先于高级配置数量
- 数据处理规则清晰稳定优先于复杂灵活
- 结构可持续扩展，便于后续接入真实业务规则

当前这份设计文档对应的是“第一版最小成品”，不是最终完整产品规格。
